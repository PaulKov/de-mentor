# Case Study: Marketplace Revenue Analytics

## Контекст

Маркетплейс строит ежедневную аналитику по выручке, регионам, категориям товаров и платежному статусу. Команда перенесла часть аналитики из PostgreSQL в Greenplum, но первый отчет по регионам начал работать нестабильно: иногда быстро, иногда заметно медленнее после полной загрузки факта продаж.

## Бизнес-Вопросы

- Какая выручка по регионам за последние 90 дней?
- Какие категории дают максимальную маржу?
- Как меняется доля отмен и возвратов?
- Какие клиенты формируют большую часть оборота?

## Инженерный Конфликт

Разработчик выбрал `status` как distribution key, потому что это популярный фильтр. В Greenplum это опасное решение: у `status` низкая cardinality, и большая часть строк может оказаться на одном сегменте.

## Миссия Ученика

1. Доказать, что проблема воспроизводится.
2. Найти skew через `gp_segment_id`.
3. Найти `Redistribute Motion` в плане запроса.
4. Предложить физическую модель для факта продаж.
5. Защитить решение как дата инженер на design review.

## Ожидаемый Профессиональный Ответ

Хороший ответ не ограничивается "поменять ключ". Он объясняет:

- grain факта;
- основные query patterns;
- почему `status` плох как distribution key;
- почему `customer_id` лучше для join с customer dimension;
- где нужен partitioning по дате;
- какие риски остаются после исправления.
